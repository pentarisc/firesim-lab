# `fslab sim fpga` ‚Äî Run Pipeline Guide

`fslab sim fpga` executes a built bitstream on a real AWS F2 FPGA host
and streams the driver's output back to you (foreground) or runs it in
the background under nohup with a local stamp tracking the in-flight
state (detached).

This guide assumes you have:

  * A successful `fslab build fpga` for the current project ‚Äî the
    driver binary lives at `build/fpga/cl_{quintuplet}/driver/FireSim-f2`.
  * An AGFI id, either from your own build's `publish.aws_afi` output
    or from a prior published artifact you have entitlement on.
  * The run-host IAM role from [aws-setup-run.md](aws-setup-run.md)
    (or an `external` host where you've installed AWS credentials
    yourself).

## fslab.yaml ‚Äî `target.run` block

Add a `target.run` block alongside `target.build` in fslab.yaml. Two
axes mirror the build side, with one extra level of nesting:
`runner_args` lives inside an `fpga_slot:` block which itself lives
inside `host:`.

```yaml
target:
  platform: f2
  clock_period: "1.0"
  fpga_sim: xsim
  build:
    # ... existing build config ...
  run:
    host:
      type: ec2_launch
      region: us-west-2
      aws_profile: fslab-run
      iam_instance_profile: fslab-fpga-runner
      lifecycle: spot_one_time
      ami_id: ami-082c5db2375456e1a
      instance_type: f2.4xlarge
      ssh_key: ~/.ssh/fslab_ed25519
      ssh_user: ubuntu
      key_name: firesim-lab
      fpga_slot:
        id: 0                            # single-slot today; must be 0
        runner_args:
          # max_cycles: 1000000          # optional cycle cap
          tracing: false
          autocounter: false
          verify_hash: IF_PRESENT        # YES | NO | IF_PRESENT (default)
          payloads: []                   # see "Payload axis" below
          result_files: []               # see "Payload axis" below
          extra_driver_flags: []         # escape hatch for raw driver flags
    artifact_source:
      type: aws_afi
      agfi: agfi-0123456789abcdef0
```

The full template is generated by `fslab new` (commented out by
default) ‚Äî see [fslab.yaml.j2](../fslab-cli/fslab/templates/fslab.yaml.j2)
for the canonical example.

The `fpga_slot:` block is single-instance today; placing it under
`host:` is the forward-compatible scaffold for multi-host / multi-slot
support, which will refactor to
`hosts: [ { ..., slots: [ { id, runner_args }, ... ] }, ... ]` with the
inner shape unchanged.

### `runner_args` reference

| Field | Type | Purpose |
|---|---|---|
| `max_cycles` | `int?` | Cap the simulation at N target cycles before the driver exits. `None` ‚áí run to natural termination (target `poweroff`, fatal error, etc.). |
| `tracing` | `bool` | Reserved for TracerV port selection; today this flag does not change driver invocation. Per-port gating lands later. |
| `autocounter` | `bool` | Reserved for autocounter CSV emission; same status as `tracing`. |
| `payloads` | `list[Payload]` | Files staged into the remote slot dir alongside the driver. Addressable from `extra_driver_flags` by `remote_name`. See "Payload axis" below. |
| `result_files` | `list[ResultFile]` | Files produced by the driver to pull back into `run/fpga/results/<ts>/`. See "Payload axis" below. |
| `verify_hash` | `enum` | `YES` / `NO` / `IF_PRESENT` (default). Toggles sha256 verification against `payloads/SHA256SUMS`. |
| `extra_driver_flags` | `list[str]` | Verbatim flags appended to the driver invocation. The framework does no validation. |

`max_cycles`, `extra_driver_flags`, the payload axis, and `verify_hash`
are wired through end-to-end. The other fields are recognised and
persisted in the run stamp for forensics; their driver-side semantics
will be filled in as needed.

## Payload axis

The payload axis lets you ship user files (loadmem binaries, kernels,
ROM images, NIC traces, ‚Ä¶) to the run host alongside the driver and
addresses them from `extra_driver_flags`. Symmetric `result_files`
pull driver-produced files back.

### Project layout

`fslab new` scaffolds an empty `payloads/` dir at the project root,
sibling to `user_rtl/`. The generated `.gitignore` excludes the
directory's contents but **keeps `payloads/SHA256SUMS` committable** ‚Äî
the manifest is the reproducibility contract; the binaries themselves
are not version-controlled.

```
<project>/
‚îú‚îÄ‚îÄ payloads/
‚îÇ   ‚îú‚îÄ‚îÄ SHA256SUMS          # optional; if present, used for verification
‚îÇ   ‚îú‚îÄ‚îÄ dhrystone.bin
‚îÇ   ‚îî‚îÄ‚îÄ initrd.cpio
‚îú‚îÄ‚îÄ user_rtl/
‚îÇ   ‚îî‚îÄ‚îÄ ...
‚îî‚îÄ‚îÄ fslab.yaml
```

### YAML shape

```yaml
target:
  run:
    host:
      # ... type / region / ssh_user / etc. ...
      fpga_slot:
        id: 0
        runner_args:
          verify_hash: IF_PRESENT
          payloads:
            - path: payloads/dhrystone.bin   # project-relative or absolute
              remote_name: dhrystone.bin     # optional; defaults to basename(path)
            - path: payloads/initrd.cpio
          result_files:
            - remote_path: dump.bin          # produced under remote slot dir
              local_name: dump.bin           # optional; defaults to basename(remote_path)
          extra_driver_flags:
            - "+loadmembin=dhrystone.bin"    # references remote_name
            - "+initrd=initrd.cpio"
```

`payloads[*]`:
  * `path` ‚Äî local source (project-relative or absolute).
  * `remote_name` ‚Äî filename inside the per-slot remote dir. Defaults
    to `basename(path)`. The driver `cd`s into the slot dir before
    exec, so `+loadmembin=dhrystone.bin` resolves correctly without
    any absolute-path machinery.

`result_files[*]`:
  * `remote_path` ‚Äî relative to the per-slot remote dir.
  * `local_name` ‚Äî filename under `run/fpga/results/<ts>/`. Defaults
    to `basename(remote_path)`. Missing files at pull time produce a
    warning, not a fatal error.

### `verify_hash` and `payloads/SHA256SUMS`

The manifest is a standard `sha256sum -c` compatible file at
`<project>/payloads/SHA256SUMS`:

```
b94d27b9934d3e08a52e52d7da7dabfac484efe37a5380ee9088f7ace2efcde9  dhrystone.bin
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855  initrd.cpio
```

Generate it with `cd payloads && sha256sum * > SHA256SUMS` (excluding
the manifest itself, since it can't be in its own check). Commit it.

`verify_hash` semantics:

| Value | Manifest present | Manifest absent |
|---|---|---|
| `YES` | Verify; fail on mismatch. | Fail at config-load time. |
| `IF_PRESENT` (default) | Verify; fail on mismatch. | Skip; warn once. |
| `NO` | Skip. | Skip. |

Verification runs in **two places**, both gated by `verify_hash`:

  1. **Locally**, before any host work ‚Äî catches a stale local manifest
     or a corrupted local payload before you pay for an EC2 instance.
  2. **On the remote**, after upload, before AGFI clear ‚Äî catches
     in-flight corruption and guarantees the FPGA host runs the bytes
     you intended.

Failure in either location is fatal:
  * Local failure aborts before host acquisition with a clear error.
  * Remote failure surfaces in `result.yaml` as
    `failure.stage: hash_verify`. The FPGA is never programmed when
    this happens ‚Äî no bitstream load is wasted on a known-corrupt
    payload set.

### Reserved remote names

The following are reserved by the framework and rejected at config-load
time if used as `remote_name` or `remote_path`:

`driver.log`, `result.yaml`, `remote_run_f2.sh`, `SHA256SUMS`, `pid`,
`run.yaml`, `.fslab`, and the driver binary basename
(`<driver_name>-<platform_id>`).

Pick a different name; framework artefacts are pulled back
automatically.

### Forensics

Every run's `result.yaml` records a `payloads:` block:

```yaml
payloads:
  - remote_name: dhrystone.bin
    sha256: b94d27b9934d3e08a52e52d7da7dabfac484efe37a5380ee9088f7ace2efcde9
    size_bytes: 12345
  - remote_name: initrd.cpio
    sha256: e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
    size_bytes: 67890
```

`sha256` is null when no manifest was in scope; `size_bytes` always
present. The block matches between foreground and detached runs.

## Ephemeral vs. managed-reuse run hosts

The `ec2_launch` host model has two sub-modes, selected by the presence
of `host.instance_id`. The build pipeline exposes the same two modes;
the run side behaves identically.

### Ephemeral (default ‚Äî `instance_id` unset)

The example above is ephemeral. Every `fslab sim fpga` invocation calls
`RunInstances` with the configured `lifecycle` (`spot_one_time` or
`on_demand`), waits for SSH, runs the simulation, and **terminates** the
instance on release. Right default for one-off runs.

The ephemeral-mode fields (`ami_id`, `instance_type`, `lifecycle`,
`subnet_id`, `key_name`) are only consulted in this mode.

### Managed reuse (`instance_id` set)

Set `host.instance_id` to point at a pre-existing EC2 instance that
fslab should reuse instead of launching a new one:

```yaml
target:
  run:
    host:
      type: ec2_launch
      region: us-west-2
      aws_profile: fslab-run
      iam_instance_profile: fslab-fpga-runner   # see note below
      instance_id: i-0abc1234567890abc           # opts in to managed reuse
      ssh_key: ~/.ssh/fslab_ed25519
      ssh_user: ubuntu
      remote_platform_path: /opt/aws-fpga       # required for ec2_launch
    artifact_source:
      type: aws_afi
      agfi: agfi-0123456789abcdef0
```

The provider looks up the named instance and behaves as follows:

| Found state | What fslab does at request | What fslab does at release |
|---|---|---|
| `stopped` | `StartInstances`, wait for SSH, run simulation. | `StopInstances` ‚Äî restores the instance to `stopped`. |
| `running` | Connect over SSH; run simulation. Logs a warning that it will **not** touch state on release. | Leave running. |
| `pending` / `stopping` | Wait for the transient state to settle, then re-check and proceed as above. | As above. |
| terminal (`terminated`, `shutting-down`) | Fail at request ‚Äî the instance cannot be used. | n/a |

In short: **"start when stopped" and "leave as found"**, exactly mirroring
the build side. If you started the instance manually before invoking
`fslab sim fpga`, fslab assumes another process or user owns its
lifecycle and won't shut it down behind your back. If fslab started it
from `stopped`, fslab is the owner and stops it on release.

Ephemeral-mode-only fields (`ami_id`, `instance_type`, `lifecycle`,
`subnet_id`, `key_name`) are ignored when `instance_id` is set ‚Äî the
instance already exists, so launch-time configuration is moot.

The same lifecycle semantics flow through detached mode: the captured
"start-action" (`launched` / `started` / `connected`) is serialised into
the run stamp's `cleanup` block, and `fslab monitor run` /
`fslab abandon run` honour it without re-deriving anything from
`fslab.yaml`. You can edit `instance_id` in `fslab.yaml` mid-flight
without breaking cleanup of the original instance.

### IAM caveat for managed reuse

fslab does **not** modify the IAM instance profile of a pre-existing
instance. The instance must already have a role attached that grants the
two actions `fpga-load-local-image` needs:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["ec2:DescribeFpgaImages", "ec2:AssociateFpgaImage"],
    "Resource": "*"
  }]
}
```

The `iam_instance_profile` field is still required in fslab.yaml ‚Äî it is
recorded in the run stamp (so cleanup logs are unambiguous about which
role *should* have been attached) and validated for shape, but no AWS
call is made to attach or change it. If your pre-existing instance has a
different profile name attached than the one in fslab.yaml, the
simulation will still succeed as long as the attached role grants the
two actions above; the mismatch is otherwise informational.

See [aws-setup-run.md](aws-setup-run.md) for the one-time IAM role
creation steps; the same role works for both ephemeral and managed-reuse
modes.

## Foreground mode (default)

```bash
fslab sim fpga
```

1. Resolves `target.run` from fslab.yaml.
2. Locally verifies `payloads/SHA256SUMS` (if `verify_hash` is in scope).
   A mismatch aborts before host acquisition ‚Äî no EC2 spend.
3. Requests a run host via the configured `host.type` (launches an EC2
   instance for `ec2_launch`, SSHes to a pre-provisioned box for
   `external`).
4. Uploads the local driver binary + every configured payload + (when
   in scope) `SHA256SUMS` to the per-slot dir on the remote.
5. On the remote, `sha256sum -c SHA256SUMS` runs in the slot dir
   (gated by `verify_hash`). A mismatch surfaces as
   `failure.stage: hash_verify` and aborts before any FPGA mgmt
   command ‚Äî no bitstream load wasted.
6. `sudo fpga-clear-local-image -S 0 -A` then
   `sudo fpga-load-local-image -S 0 -I <agfi> -A`.
7. Execs the driver under a pty so your terminal becomes the simulated
   UART ‚Äî keystrokes go to the target, target output streams back.
8. On driver exit (clean termination, fatal error, Ctrl+C):
     * `driver.log` and every configured `result_files` entry are
       rsynced back to `run/fpga/results/<timestamp>/`.
     * A local `result.yaml` is written with run_id, exit_code,
       artifact_source, started_at/finished_at, and the payload
       forensics block.
     * `sudo fpga-clear-local-image -S 0 -A` runs as teardown.
     * The host is released per its lifecycle (terminate / stop /
       leave alone).

Foreground is the right default for short interactive runs. Ctrl+C
tears down cleanly.

## Detached mode

```bash
fslab sim fpga --detach
```

Same staging + AGFI-load flow (including local + remote hash
verification), but the driver is launched in the background under
`nohup`. The local CLI writes a stamp at `run/fpga/.fslab/run.yaml` and
exits with the run_id. The remote keeps running across SSH disconnects,
laptop sleep, etc.

Use detached when the workload is long (Linux boot + SPEC, multi-hour
benchmarks) or you need to step away.

### `fslab monitor run`

```bash
fslab monitor run
```

Reads the local stamp, SSHes to the recorded run host, verifies the
remote stamp's `run_id` matches, then:

  * **Wrapper still alive:** tails `driver.log` over SSH. Ctrl+C
    detaches cleanly ‚Äî the run continues. Re-run `fslab monitor run`
    to re-attach.
  * **Wrapper has exited:** pulls `driver.log`, every `result_files`
    entry, and `result.yaml` back into a fresh
    `run/fpga/results/<timestamp>/`, runs the registered cleanup
    (terminate / stop the EC2 instance), writes a terminal stamp,
    and prints a summary.
  * **Terminal stamp already on disk:** prints the summary and exits.

The local `result.yaml` is enriched with a `payloads:` forensics block
on the monitor side, so the layout is uniform with foreground runs.

### `fslab abandon run`

```bash
fslab abandon run
```

Discards the current in-flight detached run:

  * Runs `cleanup_remote(stamp.cleanup)` against the recorded provider
    (terminate / stop / leave-alone per the captured lifecycle).
  * On cleanup success: deletes `run/fpga/.fslab/` and
    `run/fpga/staging/`.
  * On cleanup failure (e.g. expired SSO): the local stamp is
    **preserved** so you can retry ‚Äî `fslab abandon run` is idempotent.

Prior results under `run/fpga/results/` are **never** removed by
abandon ‚Äî every run's directory is append-only forensic history. Use
`fslab clean` or remove manually if you want them gone.

## Local file layout

```
<project>/
‚îú‚îÄ‚îÄ build/
‚îÇ   ‚îî‚îÄ‚îÄ fpga/
‚îÇ       ‚îî‚îÄ‚îÄ cl_<quintuplet>/
‚îÇ           ‚îî‚îÄ‚îÄ driver/
‚îÇ               ‚îî‚îÄ‚îÄ FireSim-f2         # built by `fslab build fpga`
‚îú‚îÄ‚îÄ payloads/                          # user-provided run-time inputs
‚îÇ   ‚îú‚îÄ‚îÄ SHA256SUMS                     # optional manifest (committable)
‚îÇ   ‚îî‚îÄ‚îÄ *.bin / *.elf / *.cpio / ‚Ä¶     # gitignored by default
‚îî‚îÄ‚îÄ run/
    ‚îî‚îÄ‚îÄ fpga/
        ‚îú‚îÄ‚îÄ .fslab/
        ‚îÇ   ‚îî‚îÄ‚îÄ run.yaml               # in-flight stamp (--detach only)
        ‚îú‚îÄ‚îÄ staging/
        ‚îÇ   ‚îî‚îÄ‚îÄ remote_run_f2.sh       # rendered wrapper (--detach only)
        ‚îî‚îÄ‚îÄ results/
            ‚îî‚îÄ‚îÄ <YYYYMMDD-HHMMSS>/     # one dir per invocation
                ‚îú‚îÄ‚îÄ driver.log
                ‚îú‚îÄ‚îÄ result.yaml
                ‚îî‚îÄ‚îÄ <result_files...>  # per runner_args.result_files
```

`run/fpga/staging/` is wiped at the start of every detached run. The
wrapper is re-rendered each time so project-static substitutions
(platform_id, project_name) always reflect the current fslab.yaml.

## Remote file layout

Per-slot dir on the run host (under `remote_platform_path` from
`target.run.host`):

```
<remote_slot_dir>/
‚îú‚îÄ‚îÄ FireSim-f2                 # uploaded driver
‚îú‚îÄ‚îÄ <payload files...>         # uploaded per runner_args.payloads
‚îú‚îÄ‚îÄ SHA256SUMS                 # uploaded when verify_hash is in scope
‚îú‚îÄ‚îÄ remote_run_f2.sh           # uploaded wrapper (--detach only)
‚îî‚îÄ‚îÄ .fslab/
    ‚îú‚îÄ‚îÄ run.yaml               # remote stamp (matches local run_id)
    ‚îú‚îÄ‚îÄ driver.log             # tee'd driver stdout+stderr
    ‚îú‚îÄ‚îÄ result.yaml            # written by wrapper EXIT trap
    ‚îî‚îÄ‚îÄ pid                    # wrapper PID, for liveness checks
```

## In-flight guard

Only one detached run per project may be in flight at a time. Starting
a second `fslab sim fpga --detach` while a stamp exists is refused with
a hint to run `fslab monitor run` (to attach) or `fslab abandon run`
(to discard). Foreground runs do not write a stamp and don't
participate in this guard.

## Common pitfalls

  * **`Driver binary not found`** at run start. You haven't run
    `fslab build fpga` for the current quintuplet ‚Äî or the build
    finished without producing the driver. The expected path is in
    the error message.

  * **`[PAY-01] payloads[N].path=... does not resolve to an existing
    file`**. The path is missing or misspelled. Paths are project-
    relative by default; absolute paths are also accepted.

  * **`[PAY-04] verify_hash: YES requires a sha256 manifest at
    payloads/SHA256SUMS`**. Either generate the manifest
    (`cd payloads && sha256sum * > SHA256SUMS`) or relax `verify_hash`
    to `IF_PRESENT` / `NO`.

  * **`[PAY-05] payloads/SHA256SUMS is missing entries for: [...]`**.
    A payload listed in `runner_args.payloads` doesn't appear in the
    manifest. Append it: `sha256sum <file> >> payloads/SHA256SUMS`.

  * **`failure.stage: hash_verify` in result.yaml.** The remote
    `sha256sum -c` mismatched. Usually means the local manifest is
    stale (regenerate it), or a file in `payloads/` was modified
    after the manifest was created. The FPGA was never programmed,
    so no driver-side damage to investigate.

  * **AGFI fails to load** with `AccessDeniedException`. The run
    host's IAM role is wrong. See
    [aws-setup-run.md#troubleshooting](aws-setup-run.md#troubleshooting).

  * **AGFI fails to load** with `InvalidFpgaImageID.NotFound`. The
    AGFI was registered in a different region. Either replicate it
    (`target.build.publish.copy_to_regions`) or pin the run host to
    the build region.

  * **Detached run hangs at "verify-started timed out"**. The wrapper
    failed to start on the remote ‚Äî usually a missing
    `fpga-load-local-image` (non-FPGA AMI), passwordless-sudo
    misconfig, or a malformed AGFI. SSH into the host and inspect
    `<remote_slot_dir>/.fslab/driver.log` directly. After resolving,
    run `fslab abandon run` and retry.

  * **Spot interruption mid-run.** Currently surfaced, not recovered.
    `fslab monitor run` will report the connection failure; run
    `fslab abandon run` to clean up the stamp and retry from scratch.

  * **`instance_id` points at a terminated instance.** The provider
    refuses with `Instance ... is in terminal state 'terminated'`. EC2
    instance IDs are not reusable after termination ‚Äî drop the
    `instance_id` field to fall back to ephemeral mode, or set it to
    a still-usable instance.
