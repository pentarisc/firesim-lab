# Template Rendering

This page documents the **rendering mechanism** — how the Python side loads,
configures, contextualises, and writes Jinja2 templates, and the protections
around overwriting generated files. For a catalogue of the individual template
files (what each one is and the variables it consumes) see
{doc}`/developer/jinja-templates`. For the validated models that feed the
context, see {doc}`schemas`.

Templates are the **single source of truth** for everything `fslab` generates:
the Chisel shim (`Top.scala`, `DUT.scala`, `Config.scala`), `build.sbt` /
`project/plugins.sbt`, `CMakeLists.txt`, the C++ driver, the `user_rtl/README`,
and the per-platform remote-build wrapper. Change what is generated by changing
a template plus (if it needs new data) the context builder — never by
post-processing rendered output in Python.

## The rendering pipeline at a glance

```text
load_and_validate()           → (FSLabConfig, MasterRegistry)
        │
_build_template_context()     → flat dict  (commands/context.py)
        │
_render_templates()           → render_plan: {template → output path}
        │                          PackageLoader("fslab", "templates")
        ▼
   per-template: get_template → render(**ctx) → write_text
```

`_render_templates()` in `commands/build.py` is the engine; `context.py` is the
data prep. Everything below expands those two.

## The Jinja2 environment

Templates are packaged inside the wheel under `fslab/templates/` and loaded with
a `PackageLoader`, so they resolve regardless of the user's CWD:

```python
from jinja2 import Environment, PackageLoader, select_autoescape

env = Environment(
    loader=PackageLoader("fslab", "templates"),
    autoescape=select_autoescape(enabled_extensions=()),   # OFF — we emit code, not HTML
    keep_trailing_newline=True,
)
```

Two settings matter for template authors:

- **Autoescape is disabled.** Output is Scala/C++/CMake/YAML, not HTML — HTML
  escaping would corrupt it. Never rely on escaping for safety here.
- **`keep_trailing_newline=True`**, and `trim_blocks` / `lstrip_blocks` are
  **not** set. So templates manage their own whitespace explicitly with the
  `{%- … -%}` / `{{- … -}}` trim markers. This is why the `.scala.j2` templates
  are dense with `-` on their block tags — it is deliberate, not noise. When you
  edit a template, mind the whitespace markers or you will get stray blank lines
  in generated Scala.

The `fslab init` path constructs its own small `Environment` (same loader and
settings) to render just `fslab.yaml.j2` — that is the only render that happens
*outside* the main pipeline, because at `init` time there is no validated config
yet (see {doc}`commands`).

## Building the context (`commands/context.py`)

`_build_template_context(config, registry)` flattens the two validated objects
into one plain dict. A field is **invisible to every template until it is added
here** — this is the seam between the schema layer and the templates.

The context carries several shapes:

- **Scalars / paths** — `project_name`, `package_name`, `fslab_top`,
  `config_class`, `platform`, `clock_period`, `driver_name`, `gen_dir`,
  `gen_file_basename`, the `firesim_root` / `firesim_lab_root` / `platforms_root`
  toolchain roots (defaulted when unset), and `quintuplet`
  (`{project_name}-{platform}`, the unique build id).
- **Derived design fields** — `clock_port` / `reset_port` / `enable_port` are
  found by scanning `design.blackbox_ports` for the values `in clock` / `in
  reset` / `in enable`; `verilog_files` / `verilog_file_names` come from
  `design.sources`.
- **C++ build lists** — `cxx_flags` split into tokens, `link_libs` (user libs +
  the always-required system libs `pthread`/`gmp`/`rt`/`stdc++`, de-duped,
  first-occurrence order), and the source-file lists (`user_cc_files`,
  `fslab_bridge_cc_files`, `custom_bridge_cc_files`, and the header equivalents).
- **Resolved registry entries** — `platform_cfg`, `metasim_cfg`, `fpgasim_cfg`
  are the `PlatformEntry` / `MetaSimEntry` / `FpgaSimEntry` objects themselves,
  passed through so `CMakeLists.txt.j2` can read `platform_cfg.extra_libs`,
  `metasim_cfg.cmake_targets`, etc. directly.
- **Wrapper fields** — for the F2 remote-build script (`fpga_frequency`,
  `build_strategy`, `s3_bucket_base`, `aws_region`, …), pulled from
  `target.build` and only meaningful for the relevant publish/host types.

### The three bridge shapes

Bridges appear in the context in **three shapes**, because three templates
iterate them three different ways. Understanding this is the key to the bridge
templates:

| Key | Shape | Used by | Why |
|---|---|---|---|
| `unique_bridges` | `dict[type_id, BridgeInstance]` | `Top.scala.j2`, `DUT.scala.j2` | one import block per bridge **type** |
| `used_bridges` | `list[BridgeInstance]` (unique types) | `driver.cc.j2`, CMake source lists | iterate unique types in order |
| `instances` | `list[BridgeInstance]` (per instance) | `Top.scala.j2`, `DUT.scala.j2` | one port/wiring snippet per **instance** |

Each `BridgeInstance` is a `SimpleNamespace` (so templates use dot access)
combining the registry `BridgeEntry` fields (`id`, `label`, `cpp_type`,
`cpp_headers`, `cpp_sources`, `input_ports`, `output_ports`, and the
`scala_templates` sub-paths) with — on the per-instance entries only — the
project `BridgeConfig` fields `name`, `port_map`, and `params`. So in a wiring
snippet you can write `{{ instance.name }}`, `{{ instance.port_map.txd }}`, and
`{{ instance.params.baud_rate.value }}`.

## The render plan

`_render_templates()` maps each template to its output path:

```python
render_plan = {
    "build.sbt.j2":          project_root / "build.sbt",
    "plugins.sbt.j2":        project_root / "project" / "plugins.sbt",
    "CMakeLists.txt.j2":     project_root / "CMakeLists.txt",
    "Top.scala.j2":          project_root / "src/main/scala" / f"{fslab_top}.scala",
    "DUT.scala.j2":          project_root / "src/main/scala" / f"{fslab_top}BlackBox.scala",
    "Config.scala.j2":       project_root / "src/main/scala" / "Config.scala",
    "driver.cc.j2":          project_root / "src/main/cc" / f"{driver_name}.cc",
    "user_rtl_readme.md.j2": project_root / "user_rtl" / "README.md",
}
```

Output file *names* are themselves derived from the context (`fslab_top`,
`driver_name`). The plan is **conditionally extended** per platform — only F2
ships a wrapper template today:

```python
if config.target.platform == "f2":
    render_plan["remote_build/f2.sh.j2"] = project_root / "scripts" / "remote_build_f2.sh"
```

The wrapper lands under `scripts/` (outside `build/`, so it survives `fslab
clean`) and is re-uploaded on every `fslab build fpga`. When you add a template,
add its `render_plan` entry here; when you add a platform with its own wrapper,
extend this conditional.

Rendering loops the plan, calling `env.get_template(name)` then
`tmpl.render(**ctx)`. It is defensive about template problems — `TemplateNotFound`
and `TemplateSyntaxError` are caught, logged as warnings, and mark the render
unsuccessful rather than crashing — but a successful render writes every file
and records success.

## Sub-template inclusion: how bridges compose

The bridge `.j2` files under `templates/bridges/<id>/` are **not** in the render
plan. They are pulled in by the top-level Scala/C++ templates via `{% include %}`,
using paths carried on each `BridgeInstance`:

```jinja
{# DUT.scala.j2 — one port snippet per instance #}
{% for instance in instances %}
{% if instance.scala_templates.ports %}
{% include instance.scala_templates.ports %}
{% endif %}
{% endfor %}
```

Because the include paths (`bridges/uart/ports.scala.j2`, …) are resolved by the
same `PackageLoader`, they are relative to the `templates/` root. The set of
per-bridge sub-templates:

| Sub-template | Included by | Emits |
|---|---|---|
| `scala_templates.dut_imports` | `DUT.scala.j2` (per type) | DUT-level Scala imports |
| `scala_templates.top_imports` | `Top.scala.j2` (per type, optional) | Top-level Scala imports |
| `scala_templates.ports` | `DUT.scala.j2` (per instance) | IO bundle port declarations |
| `scala_templates.wiring` | `Top.scala.j2` (per instance) | bridge ↔ `dut.io` wiring |
| `cpp_template` | `driver.cc.j2` (per type, inside the sim loop) | per-bridge driver tick logic |

A useful idiom in `Top.scala.j2` / `driver.cc.j2`: the included snippet is
captured into a variable and re-indented so it nests correctly:

```jinja
{%- set content %}
{% include instance.scala_templates.wiring %}
{%- endset -%}
{{ content | indent(4, true) }}
```

So a bridge author writes the snippet at column 0 and the host template handles
indentation. When you add a bridge, you provide these sub-templates and point
the registry `scala_templates` / `cpp_template` paths at them — see
{doc}`extending`.

## Registry-driven generation: the CMake / Makefile case

`CMakeLists.txt.j2` is worth calling out because it shows the framework's
"no hard-coded platform knowledge" rule at its most thorough. It renders four
independent build sub-trees (`driver/`, `metasim/`, `fpgasim/`, `fpga/`) entirely
from `platform_cfg` / `metasim_cfg` / `fpgasim_cfg`. To add a simulator or a
platform you add a registry entry — you do **not** edit the template.

It also generates *Makefiles inside CMake* via `file(WRITE ...)`, which forces a
two-level escaping discipline you must respect if you edit it:

- `${CMAKE_VAR}` — CMake expands at configure time.
- `\$(make_var)` — passed through; **Make** expands at build time.
- `$$$$ORIGIN` — survives the CMake→Make→shell→linker chain to reach the linker
  as `$ORIGIN`.

This is why the `extra_include_dirs` of a `MetaSimEntry` must use Makefile path
syntax while a `PlatformEntry`'s use CMake syntax — the schema layer enforces the
split (`REG-11` vs `REG-11m`, see {doc}`schemas`) precisely so the right one
lands in the right generated file.

## Protecting hand-edited output: the hash gate

Generated files are owned by `fslab`, but users sometimes edit them. Two
mechanisms in `utils.state.StateManager` (see {doc}`commands`) prevent silent
data loss:

1. **Config-hash skip.** If `fslab.yaml` + every registry file hash to the same
   value as last time, generation is skipped entirely (unless `--force`). This
   is why `build`/`sim` can call generate unconditionally.
2. **Per-file modification guard.** Before overwriting, `check_user_modifications`
   compares each output file against the hash stored when it was last generated.
   If the user changed (or deleted) a generated file, `_render_templates` returns
   the change set and `generate` **refuses** with a "Detected changes to
   bootstrapped files" error listing each `modified` / `missing` path — unless
   `--force`. After a successful render, `compute_generated_files_state` records
   the new hashes back into `state.json`.

So the contract for a template author is: the rendered file is yours to
regenerate, the user's edits are protected by default, and `--force` is the
documented override.

## Checklist for changing what gets generated

- New output file → add a template under `templates/` **and** a `render_plan`
  entry in `_render_templates`.
- New data the template needs → add it to `_build_template_context` first; it is
  invisible otherwise.
- New per-bridge behaviour → add the sub-template under `templates/bridges/<id>/`
  and reference it from the registry entry (`scala_templates` / `cpp_template`),
  not from the render plan.
- New platform wrapper → extend the per-platform conditional in the render plan.
- Mind whitespace (`{%- -%}`) — autoescape is off and block-trimming is not
  enabled.
- Keep verbatim-emit registry fields (`cmake_fragment` / `makefile_fragment`)
  free of Jinja2 markers — the schema layer rejects them (`REG-13`/`REG-14`).
- Update the catalogue in {doc}`/developer/jinja-templates` when you add or
  rename a template file.
